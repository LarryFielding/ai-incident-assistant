0. Designed the AI analysis domain — COMPLETED
----------------------------------------------
The request, response, enums, validations, and JSON conventions were designed before implementation.

Final request shape:

{
  "title": "Database timeout in payment service",
  "description": "Users report intermittent failures during checkout",
  "raw_logs": "java.sql.SQLTimeoutException: query timed out",
  "service_name": "payment-service",
  "environment": "PROD",
  "environment_name": "payments-prod-us-east-1",
  "incident_occurred_at": "2026-05-28T03:15:00Z"
}

Final response shape:

{
  "summary": "...",
  "severity": "HIGH",
  "category": "DATABASE",
  "possible_root_cause": "...",
  "suggested_actions": [],
  "postmortem_draft": "..."
}

Important domain decisions:
- The incident-service reports incidents to the AI service.
- The AI service is responsible for categorizing the incident.
- The request should not contain a final incident category.
- incident_occurred_at was added because the timestamp should exist at incident header level, not only inside raw logs.
- environment is a controlled enum such as PROD, DEV, QA, STAGING, LOCAL.
- environment_name is optional and can store the actual deployment/cluster/region name, such as payments-prod-us-east-1.

Enums designed:

Environment:
- LOCAL
- DEV
- QA
- STAGING
- PROD

Severity:
- LOW
- MEDIUM
- HIGH
- CRITICAL

IncidentCategory:
- DATABASE
- NETWORK
- APPLICATION
- INFRASTRUCTURE
- SECURITY
- PERFORMANCE
- UNKNOWN

Validation rules designed:

Request:
- title: required, 5–150 chars
- description: required, 20–4000 chars
- raw_logs: optional, max 20000 chars
- service_name: required, 2–100 chars
- environment: required enum
- environment_name: optional, max 150 chars
- incident_occurred_at: required datetime, ISO-8601 preferred

Response:
- summary: required, 20–1000 chars
- severity: required enum
- category: required enum
- possible_root_cause: required, 10–1500 chars
- suggested_actions: required, 1–10 items
- postmortem_draft: required, 50–4000 chars

JSON conventions:
- Use snake_case for JSON fields.
- Use uppercase enum values.
- Use ISO-8601 dates.
- Optional fields may be null.
- Use English technical field names.

6. Added rule-based mock analysis — COMPLETED
---------------------------------------------
A mock analyzer was implemented to classify incidents using simple keyword-based rules.

The purpose is both educational and architectural:
- It teaches request -> service -> response flow.
- It avoids adding LLM complexity too early.
- It can later become a fallback if the LLM fails.
- It can also become a pre-classifier or hint generator for the LLM.

The service logic was moved out of main.py into:

app/services/incident_analysis_service.py

The endpoint now delegates to a service function.

Spring Boot analogy:
- app/api/routes.py or endpoint function ≈ Controller
- app/services/incident_analysis_service.py ≈ Service
- app/models/*.py ≈ DTOs / enums

Rule-based classification examples:

Category rules:
- database, sql, query, connection pool, timeout -> DATABASE
- unauthorized, forbidden, jwt, token, authentication -> SECURITY
- dns, network, connection refused, connection reset -> NETWORK
- memory, outofmemory, heap, cpu, latency -> PERFORMANCE
- exception, nullpointer, stack trace, error -> APPLICATION
- otherwise -> UNKNOWN

Severity rules:
- PROD + outage/down/unavailable -> CRITICAL
- PROD + DATABASE/SECURITY/INFRASTRUCTURE -> HIGH
- PROD general issue -> MEDIUM
- STAGING or QA -> MEDIUM
- DEV/LOCAL -> LOW

Manual scenarios tested successfully in Swagger:
- Database timeout -> DATABASE/HIGH
- Security/authentication issue -> SECURITY/HIGH
- Network issue in staging -> NETWORK/MEDIUM
- Critical production outage -> CRITICAL severity
