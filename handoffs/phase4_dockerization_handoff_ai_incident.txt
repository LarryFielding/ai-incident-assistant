2. Arquitectura actual validada
============================================================

Flujo HTTP síncrono conservado:

Client / Postman
   ↓
POST /api/incidents/{id}/analysis
   ↓
Spring Boot incident-service
   ↓ HTTP/1.1
FastAPI POST /analyze-incident
   ↓
LLM o rule-based analyzer
   ↓
Spring Boot recibe el resultado
   ↓
PostgreSQL: incident_analyses

Flujo Kafka asíncrono completado:

Client / Postman
   ↓
POST /api/incidents/{id}/analysis-requests
   ↓ responde 202 Accepted
Spring Boot incident-service
   ↓ publica
Kafka topic: incident.analysis.requested
   ↓ consume
Python ai-service Kafka consumer
   ↓
Pydantic valida el evento
   ↓
Lógica interna de análisis
   ↓ publica
Kafka topic: incident.analyzed
   ↓ consume
Spring Boot Kafka listener
   ↓
IncidentService
   ↓
PostgreSQL: incident_analyses

El endpoint FastAPI no participa en el flujo Kafka.
El consumer Python invoca directamente la misma lógica interna que utiliza FastAPI.

7. Decisión importante para ai-service
============================================================

Actualmente ai-service tiene dos procesos ejecutables:

1. FastAPI/Uvicorn
2. Kafka consumer

No conviene asumir que un contenedor debe ejecutar ambos procesos sin diseñarlo explícitamente.

Opciones:

Opción A — Dos contenedores con la misma imagen

ai-service-api
- Ejecuta Uvicorn.
- Expone FastAPI.

ai-service-consumer
- Ejecuta python -m app.messaging.kafka_consumer.
- No expone puerto HTTP.

Ventajas:
- Separación clara de responsabilidades.
- Escalado independiente.
- Reinicios independientes.
- Más cercano a un diseño real.

Esta es la opción recomendada.

Opción B — Un solo contenedor con dos procesos

Requeriría:
- supervisor;
- script de arranque;
- gestión de señales;
- mayor complejidad.

No se recomienda para esta fase de aprendizaje.

Arquitectura Docker recomendada:

incident-service
ai-service-api
ai-service-consumer
postgres
kafka

Aunque conceptualmente siga existiendo un solo microservicio Python, se ejecutan dos workloads diferentes usando la misma imagen.

============================================================
8. Problema de localhost dentro de Docker
============================================================

Actualmente las aplicaciones usan direcciones como:

localhost:9092
localhost:5432
localhost:8000

Dentro de un contenedor:

localhost
=
el mismo contenedor

Por tanto, incident-service no debe buscar Kafka en localhost:9092 cuando esté containerizado.

Debe usar el nombre del servicio Compose:

kafka:9092

PostgreSQL:

postgres:5432

FastAPI desde incident-service:

ai-service-api:8000

Ejemplo conceptual:

Local fuera de Docker:
KAFKA_BOOTSTRAP_SERVERS=localhost:9092

Dentro de Compose:
KAFKA_BOOTSTRAP_SERVERS=kafka:9092

La Fase 4 debe eliminar valores hardcoded y usar variables de entorno.

12. Dockerfile recomendado para ai-service
============================================================

Una sola imagen base Python puede servir para:

- ai-service-api
- ai-service-consumer

La imagen debe:

- usar una versión fija de Python;
- instalar requirements.txt;
- copiar app/;
- definir directorio de trabajo;
- usar variables de entorno;
- evitar instalar dependencias innecesarias.

En docker-compose.yml se define un command distinto para cada workload.

Ejemplo conceptual:

ai-service-api:
  command: uvicorn app.main:app --host 0.0.0.0 --port 8000

ai-service-consumer:
  command: python -m app.messaging.kafka_consumer

13. Redes y service discovery
============================================================

Docker Compose crea una red interna por defecto.

Los servicios se resuelven por nombre:

incident-service
ai-service-api
ai-service-consumer
postgres
kafka

Ejemplo:

incident-service → kafka:9092

ai-service-consumer → kafka:9092

incident-service → postgres:5432

incident-service → http://ai-service-api:8000

No es necesario publicar todos los puertos al host.

Puertos que probablemente sí deben exponerse:

incident-service:
8080:8080

ai-service-api:
8000:8000

PostgreSQL:
5432:5432 solo si se desea acceder desde herramientas locales

Kafka:
9092:9092 solo si se desea acceder desde aplicaciones fuera de Docker

El consumer Python no necesita exponer un puerto.

14. Health checks y orden de arranque
============================================================

depends_on no garantiza por sí mismo que una aplicación esté lista.

Solo puede garantizar orden de creación, salvo que se combine con health checks y condiciones compatibles.

Se recomienda agregar health checks para:

- postgres
- kafka
- ai-service-api
- incident-service

Sin embargo, no hay que intentar resolver toda la resiliencia de arranque en el primer paso.

Orden recomendado:

1. Dockerfiles funcionales.
2. Compose consolidado.
3. Conectividad entre servicios.
4. Variables de entorno.
5. Health checks.
6. Reinicios y refinamientos.

Las aplicaciones también deberían tolerar que una dependencia tarde unos segundos en estar disponible.

18. Evaluación de Fase 5 antes de nube
============================================================

La Fase 5 actual se llama:

Kafka Event Streaming (Advanced)

Gran parte de su objetivo base ya fue completado en Fase 3.1:

- producer/consumer patterns;
- comunicación asíncrona;
- serialización JSON;
- dos topics;
- múltiples servicios;
- persistencia del resultado.

Por tanto, Fase 5 no debería repetir la integración Kafka.

Debe redefinirse como:

Fase 5 — Event-Driven Reliability

Elementos potenciales:

- idempotencia;
- correlation_id;
- lifecycle PENDING/PROCESSING/COMPLETED/FAILED;
- retries;
- dead-letter topic;
- incident.analysis.failed;
- acknowledgment y offsets;
- delivery callbacks;
- control de duplicados;
- tests de integración Kafka;
- creación explícita de topics;
- schema evolution.

¿Es necesaria completa antes de nube?

No.

No es necesario implementar antes del primer despliegue:

- Schema Registry.
- Avro.
- Protobuf.
- múltiples consumidores avanzados.
- DLQ sofisticada.
- outbox pattern completo.
- procesamiento exactly-once.
- Kafka UI en producción.

Sí conviene implementar antes del despliegue público:

- manejo básico de errores del consumer;
- evitar que una excepción mate permanentemente el proceso;
- logs claros con incidentId y eventId;
- configuración por variables de entorno;
- al menos una estrategia conocida para mensajes fallidos;
- idempotencia mínima o, como mínimo, documentar el riesgo de duplicados;
- creación explícita de topics o control de auto-create;
- prueba end-to-end repetible.

Recomendación:
Hacer una Fase 5 reducida después de Dockerización, no toda la lista avanzada.

21. Recomendación sobre GCP y Kafka
============================================================

El Project Context propone GCP como destino inicial.

Antes de diseñar Fase 7 habrá que decidir cómo desplegar Kafka.

Cloud Run no es el lugar natural para ejecutar un broker Kafka persistente.

Opciones futuras:

- usar un proveedor Kafka administrado;
- usar Redpanda Cloud o Confluent Cloud;
- desplegar Kafka en una VM solo con fines de aprendizaje;
- sustituir temporalmente Kafka por Pub/Sub para una versión GCP-native;
- elegir otra plataforma para el primer despliegue.

No debe tomarse esta decisión durante Fase 4.

Fase 4 debe producir imágenes portables y una configuración Compose funcional.
