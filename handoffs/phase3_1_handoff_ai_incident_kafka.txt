6. Cliente HTTP Java → FastAPI
============================================================

Se creó AiAnalysisClient para llamar a FastAPI:

Paquete:
com.example.incident_service.integration.ai

Responsabilidad:
- Recibir AiIncidentAnalysisRequest.
- Hacer POST a /analyze-incident.
- Convertir respuesta JSON a AiIncidentAnalysisResponse.

Importante problema resuelto:
Inicialmente RestClient/JDK HttpClient intentaba enviar un upgrade a HTTP/2 h2c contra Uvicorn/FastAPI.
FastAPI mostraba:

WARNING: Unsupported upgrade request.
HEADERS: connection: Upgrade, HTTP2-Settings
upgrade: h2c
transfer-encoding: chunked
RAW BODY: vacío

Esto causaba que FastAPI respondiera 422:

{"detail":[{"type":"missing","loc":["body"],"msg":"Field required","input":null}]}

Solución:
Forzar HTTP/1.1 en AiServiceClientConfig.

AiServiceClientConfig debe usar algo equivalente a:

package com.example.incident_service.config;

import org.springframework.beans.factory.annotation.Value;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.http.client.JdkClientHttpRequestFactory;
import org.springframework.web.client.RestClient;

import java.net.http.HttpClient;
import java.time.Duration;

@Configuration
public class AiServiceClientConfig {

    @Bean
    public RestClient aiServiceRestClient(
            @Value("${ai-service.base-url}") String aiServiceBaseUrl
    ) {
        HttpClient httpClient = HttpClient.newBuilder()
                .version(HttpClient.Version.HTTP_1_1)
                .connectTimeout(Duration.ofSeconds(5))
                .build();

        JdkClientHttpRequestFactory requestFactory =
                new JdkClientHttpRequestFactory(httpClient);

        return RestClient.builder()
                .requestFactory(requestFactory)
                .baseUrl(aiServiceBaseUrl)
                .build();
    }
}

En application.yml se agregó:

ai-service:
  base-url: http://localhost:8000

AiAnalysisClient debe usar:

.post()
.uri("/analyze-incident")
.contentType(MediaType.APPLICATION_JSON)
.accept(MediaType.APPLICATION_JSON)
.body(request)
.retrieve()
.body(AiIncidentAnalysisResponse.class)

Se agregó tabla incident_analyses mediante Flyway V3.

Archivo:
src/main/resources/db/migration/V3__create_incident_analyses_table.sql

Contenido conceptual:

CREATE TABLE incident_analyses (
    id BIGSERIAL PRIMARY KEY,
    incident_id BIGINT NOT NULL,
    summary TEXT NOT NULL,
    severity VARCHAR(50) NOT NULL,
    category VARCHAR(100) NOT NULL,
    possible_root_cause TEXT,
    suggested_actions TEXT,
    postmortem_draft TEXT,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_incident_analyses_incident
        FOREIGN KEY (incident_id)
        REFERENCES incidents(id)
        ON DELETE CASCADE
);

CREATE INDEX idx_incident_analyses_incident_id
    ON incident_analyses(incident_id);

CREATE INDEX idx_incident_analyses_created_at
    ON incident_analyses(created_at);

Nota aclarada durante la fase:
- incidents.id es BIGSERIAL PRIMARY KEY.
- En PostgreSQL BIGSERIAL realmente crea una columna BIGINT con default nextval(...).
- Por eso incident_analyses.incident_id debe ser BIGINT, no BIGSERIAL.
- La tabla hija no debe generar su propio id de incidente; solo referencia el id existente.

Entidad agregada:

- Campos: summary, severity, category, possibleRootCause, suggestedActions, postmortemDraft, createdAt.

9. Problema transaccional resuelto
============================================================

Al intentar guardar IncidentAnalysis apareció:

ERROR: cannot execute INSERT in a read-only transaction

Causa:
IncidentServiceImpl tenía @Transactional(readOnly = true) a nivel de clase.
El nuevo método analyzeIncident ahora hace INSERT, por lo que heredaba readOnly=true.

Solución:
Agregar @Transactional normal al método analyzeIncident.

Regla acordada:
- Métodos solo lectura: heredan @Transactional(readOnly = true).
- Métodos que hacen INSERT/UPDATE/DELETE: llevan @Transactional.

Ejemplo conceptual:

@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class IncidentServiceImpl implements IncidentService {

    @Override
    @Transactional
    public IncidentResponse createIncident(CreateIncidentRequest request) { ... }

    @Override
    public List<IncidentResponse> getAllIncidents() { ... }

    @Override
    public IncidentResponse getIncidentById(Long id) { ... }

    @Override
    @Transactional
    public IncidentResponse updateIncidentStatus(Long id, UpdateIncidentStatusRequest request) { ... }

    @Override
    @Transactional
    public AiIncidentAnalysisResponse analyzeIncident(Long id) { ... }
}

15. Diferencia conceptual que debe explicarse en Fase 3.1
============================================================

Flujo HTTP actual:
- Síncrono.
- El cliente espera respuesta inmediata.
- Si FastAPI tarda o falla, el endpoint Java tarda o falla.
- Útil para validar contrato y debug.

Flujo Kafka futuro:
- Asíncrono.
- Cliente solicita análisis y recibe respuesta rápida tipo 202 Accepted.
- El análisis se procesa después.
- Java y Python quedan desacoplados temporalmente.
- Permite reintentos, resiliencia y escalamiento.

Cambio importante de API:
Actualmente:
POST /api/incidents/{id}/analysis
→ devuelve análisis completo.

Futuro posible:
POST /api/incidents/{id}/analysis-requests
→ devuelve 202 Accepted o un tracking id.

Luego:
GET /api/incidents/{id}/analyses
→ consulta historial de análisis guardados.
